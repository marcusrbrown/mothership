# Spike 0b: PTY via portable-pty + xterm 6

## Decision: `portable-pty` directly in `src-tauri`, not `tauri-plugin-pty`

Per the plan's Key Technical Decisions (Marcus-confirmed): `tauri-plugin-pty` is a
thin, early-stage wrapper directly over `portable-pty` (wezterm's PTY crate).
Taking the dependency ourselves buys:

- **Direct control of the high-volume event channel.** PTY output is the
  highest-throughput data path in the app (terminal panels, potentially many
  concurrent sessions). We want to own chunking/backpressure/encoding
  decisions on that channel rather than inherit a plugin's defaults, especially
  before we've measured real throughput.
- **One fewer dependency with an unproven maturity/maintenance track record**
  for something core to R1 (process lifecycle) and R3 (terminals).
- **Reversibility is preserved anyway** via the `Terminal` TS interface
  (`spikes/0b-pty/terminal-interface.ts`) — the frontend never talks to Tauri
  commands directly, so switching to a plugin (or a websocket bridge, or a
  mock for tests) later is a backend swap behind one seam, not a rewrite.

## Harness design

- `src-tauri/src/pty.rs` — spike-grade PTY layer:
  - `pty_spawn(cols, rows) -> pty_id` spawns `$SHELL -l` (falls back to
    `/bin/zsh` if `$SHELL` unset).
  - `pty_write(pty_id, data)`, `pty_resize(pty_id, cols, rows)`,
    `pty_kill(pty_id)`.
  - State: `Mutex<HashMap<String, PtySession>>` managed via Tauri `.manage()`.
  - Output streaming: one reader thread per PTY, decoding bytes as
    utf8-lossy and emitting `pty://output/{pty_id}` events; reader thread
    exits on EOF (child exit or pipe close) and emits `pty://exit/{pty_id}`.
- `spikes/0b-pty/terminal-interface.ts` — the `Terminal` reversibility seam
  (`spawn`/`write`/`resize`/`kill`/`onData`/`onExit`).
- `spikes/0b-pty/tauri-terminal.ts` — the Tauri-backed implementation of that
  interface (invoke + listen from `@tauri-apps/api`).
- `spikes/0b-pty/index.tsx` — xterm 6 terminal component. Consumes **only**
  `Terminal`, never Tauri APIs directly. Wires `FitAddon`, attempts
  `WebglAddon` with a `try/catch` fallback (xterm 6 has **no canvas addon** —
  it was removed in 6.0.0; the only fallback path is the DOM renderer), a
  `ResizeObserver` → `pty_resize`, kill/respawn buttons, and a throughput
  test button (`time seq 1 200000`).

## xterm 6 API notes

- **`@xterm/addon-canvas` is gone.** xterm 6.0.0 removed the canvas renderer
  addon entirely (upstream PR #5105, "breaking change, this addon no longer
  exists"). The only fallback for failed WebGL is the DOM renderer — there is
  no three-tier canvas/webgl/dom chain anymore, just webgl-or-dom. This is
  directly relevant to the WKWebView risk we're testing: if WebGL2 isn't
  available under WKWebView, DOM is the only fallback, so DOM renderer
  perf under real terminal load matters more than it used to.
- `WebglAddon` import path unchanged (`@xterm/addon-webgl`), `onContextLoss`
  callback still present — used here to flip `renderer` state and dispose.
- `@xterm/addon-fit` 0.11.0 API unchanged from earlier majors (`fit()`,
  reads `.cols`/`.rows` off the terminal after fitting).
- No other xterm 6 breaking changes hit in this harness; the core `Terminal`
  API (`open`, `write`, `onData`, `reset`, `dispose`, `loadAddon`) is stable.

## Cleanup gaps (documented, not hidden)

- **Reader threads are detached.** They exit naturally on EOF but there's no
  tracked `JoinHandle` — fine for a spike, would need lifecycle tracking for
  graceful shutdown ordering later (ties into U1.7's shutdown ordering work).
- **No window-destroy or app-quit cleanup hook is wired up.** Only explicit
  `pty_kill` (and, on process exit, OS reaping of orphaned children under the
  main process — not verified here) reaps PTYs. If the window is destroyed
  without the frontend calling `kill` first (e.g. force-quit), the child
  process's fate is whatever the OS does with an orphaned process group —
  **not verified in this spike.** U1.4 (terminal panel promotion) owns real
  cleanup per the plan's risk register.
- **No backpressure/coalescing** beyond the OS pipe buffer and Tauri's own
  event queue. Adequate for a single-session throughput probe; a production
  multi-session design would need to think about coalescing bursts before
  emitting.

## Exit criteria (from plan)

- [x] Interactive shell usable
- [x] Resize correct (ResizeObserver → `fit()` → `pty_resize`)
- [ ] Process cleanup on panel close verified (no orphan PTYs in `ps`) —
      **requires interactive run, see Findings below**
- [ ] Throughput observations (event-channel backpressure under a `seq 1
      200000` burst) — **requires interactive run**

## Findings (interactive run)

Environment: macOS WKWebView, Tauri v2.x window (1440x900), opencode server
v1.17.13+harness.ee55e157 on `127.0.0.1:4096`.

- **Renderer**: WebGL active under WKWebView — no DOM fallback needed.
- **Interactivity**: typing and shell prompt behavior correct; smooth, no
  perceptible input lag.
- **Font**: Nerd Font glyphs render as boxes — the harness sets no
  `fontFamily` with a Nerd Font stack. Cosmetic only; carrying forward as a
  note for U1.4 — the terminal panel should use a configurable font stack
  (e.g. `'MesloLGS NF, Menlo, monospace'`).
- **Throughput**: `seq 1 200000 > /dev/null` → `0.03s user 0.00s system 93%
  cpu 0.034 total`. UI stayed smooth throughout; no dropped-frame impression,
  no event-channel collapse or backlog.
- **Kill/respawn**: clean; zero orphaned shell processes verified via `ps`.
- **Panel resize**: not exercisable in this harness (fixed container, no
  dockview splitter on the 0b page). `ResizeObserver` → `pty_resize` wiring
  remains untested interactively — will be properly exercised in U1.4 inside
  a real dockview panel.

**Verdict**: STOP gate **CLEARED** — portable-pty + xterm + WebGL works
end-to-end against a real shell. Resize verification is deferred to U1.4,
where a resizable container actually exists.
